# AGENTS

## Purpose

`_core/web_browsing/` owns the browser overlay module and the generic `<x-browser>` browser surface element.

It contributes a Browser action to the routed onscreen menu, mounts draggable, minimizable, resizable floating browser windows through the router overlay seam, and lets any on-screen DOM, including space widgets, create a browser with plain markup such as `<x-browser src="https://example.com"></x-browser>`. This module stays self-contained: it ships the floating shell, the host-side browser-surface store, the `<x-browser>` element, the browser bridge adapters, the placeholder page, and the injected runtime path that browser iframes and desktop-native views can both use.

Documentation is top priority for this module. After any change under `_core/web_browsing/`, update this file, affected parent docs, and the matching supplemental docs under `_core/documentation/docs/` in the same session.

## Ownership

This module owns:

- `menu-item.html`: real Browser dropdown action component for the routed onscreen menu
- `window.html`: floating browser-window component mounted into the router overlay layer; it renders each popup as a generic `<x-browser controls="true">`
- `browser-element.js`: autonomous custom-element definition for `<x-browser>`, including `src` navigation, `controls="true|false"` toolbar rendering, iframe fallback creation, desktop `<webview>` creation, native-surface registration, and cleanup
- `store.js`: shared Alpine store for registered browser surfaces, unique `browser-N` id generation, popup-window geometry, focus, drag, resize, minimize, toolbar state, and browser-surface lifecycle
- `window.css`: local floating-window styling plus the shared `.space-browser*` element, toolbar, frame-shell, and resize-handle styling used by popup and inline browsers
- `browser-frame.html`: module-owned iframe placeholder page used as the first local page inside each window
- `browser-frame-protocol.js`: shared browser-bridge envelope protocol, payload normalization, and iframe-window request-response bridge factory
- `browser-frame-bridge.js`: outside-side helper that Space Agent surfaces can use to talk to a browser surface by `browser-N` id or by a DOM reference to an `<x-browser>` element or its inner browser frame, whether the current runtime uses an iframe, desktop `<webview>`, or desktop-native view
- `browser-logging.js`: shared browser-overlay logging helper that owns the app-side browser log levels, defaults renderer-side browser diagnostics to `error`, exposes the supported `debug|info|warn|error|silent` levels, and lets the top-level `space.browser.setLogLevel(...)` runtime hook raise or lower browser-specific console noise without changing the standalone harness logger
- `agent-function-availability.js`: shared agent-facing function guard for the `space.browser` runtime namespace; it resolves the current runtime from the framework `<x-context>` element while preferring the explicit packaged desktop bridge when available, owns requirement checks, and returns structured warning stubs when a guarded browser action is unavailable
- `browser-native-bridge.js`: renderer-side desktop adapter that wraps the Electron `spaceDesktop.browser` bridge in the same request-response shape used by the iframe helper
- `browser-guest-runtime.js`: renderer-side guest-runtime builder wrapped with `space.extend(...)`; it owns the default phased list of guest-page runtime scripts injected into desktop browser pages around the bridge bootstrap, and extension hooks may append more pre-bootstrap or post-bootstrap script paths there without editing the desktop host
- `browser-webview.js`: renderer-side desktop `<webview>` helpers for partition naming, immediate navigation-state reads, injection preludes, and runtime injection into desktop guest pages
- `browser-webview-bridge.js`: renderer-side request-response bridge for desktop `<webview>` guests using `ipc-message` and `send(...)` on the embedder element
- `browser-surface.js`: renderer-side `<x-browser>` lifecycle and geometry helper that creates, updates, focuses, and destroys desktop-native browser views while leaving browser sessions on the iframe fallback path
- `browser-page-content.js`: repo-owned child-page helper that simplifies the live DOM into readable markdown with stable typed reference markers such as `[link 12]`, `[button 18]`, `[image 24]`, or `[input text 30]`, keeps the current reference map for later detail lookups inside the same frame, and owns the reference-targeted `click`, `type`, `submit`, `typeSubmit`, and `scroll` actions that run against that captured map; when the desktop DOM helper is present it should ask for a content-oriented snapshot instead of the raw helper markup so `content` stays readable, does not leak synthetic helper wrapper tags or escaped helper-shell text, drops obvious serialized HTML, full-document shells, or raw `<style>/<script>` blobs that would not be useful to an agent, gives images real refs so `detail(...)` can inspect them too, must also treat generic elements with inline or framework-owned event handlers such as `onclick`, `@click`, `x-on:click`, or `v-on:click` as actionable refs instead of plain text, falls back to truncated URL text for empty links or images instead of leaving blank labels, defaults to a lean token-saving summary shape with no per-link `-> url` summaries, no quotes around reference labels, and no list bullets while preserving list indentation unless the caller explicitly opts in, may add compact state and best-effort semantic tags inside the typed box such as `[disabled muted button 5] Confirm` or `[checked checkbox 6] Email updates` unless `includeStateTags === false` or `includeSemanticTags === false` suppresses them, should surface live values or selected text for form controls when useful, must return structured action-effect payloads for ref-targeted actions so callers can tell whether the page visibly reacted, and must fall back to live DOM capture when a remote page's Trusted Types policy blocks the temporary helper-HTML reparse used for nested-frame-aware readable output
- `browser-frame-inject.js`: inside-side bridge bootstrap fetched and evaluated by the packaged desktop host or addressed through iframe `postMessage`; it owns transport setup, bridge envelopes, injected-runtime metadata, the guest runtime registry, and helper utilities that later guest runtime scripts use to register request handlers plus navigation-state and `_blank` hooks
- `ext/browser_guest_runtime/**/*.js`: plain injected guest-runtime scripts loaded into desktop browser pages after the bridge bootstrap; the current core script registers the built-in browser message handlers and later layers may add more runtime scripts through the guest-runtime builder seam instead of patching the bootstrap
- `ext/js/_core/onscreen_agent/llm.js/buildOnscreenAgentTransientSections/end/open-browsers.js`: browser-owned transient-context hook that lists currently open browser surfaces for the onscreen agent only when at least one browser surface exists
- `ext/js/_core/onscreen_agent/llm.js/buildOnscreenAgentTransientSections/end/last-interacted-browser-content.js`: browser-owned transient-context hook that remembers the last browser surface targeted by agent interaction helpers or direct browser-surface focus and, during prompt construction only, loads fresh simplified page content for that browser if it is still open
- `ext/html/_core/onscreen_menu/items/browser.html`: thin routed menu-item adapter
- `ext/html/page/router/overlay/end/browser-window.html`: thin routed overlay adapter

## Local Contracts

- this module must mount only through `_core/onscreen_menu/items` and `page/router/overlay/end`; do not hardcode it into `_core/onscreen_menu/` or `_core/router/`
- the routed menu action is owned here through `_core/onscreen_menu/items` with `data-order="250"`
- the routed menu action must open a new floating browser window each time it is clicked without changing the current route
- `window.html` should export `browser:open` through `<x-context>` while at least one registered browser surface is open, including inline `<x-browser>` instances outside popup windows, so onscreen browser skills can auto-load from normal skill metadata
- floating windows are viewport-fixed, draggable from their compact title bar, minimizable and closable from their header controls, resizable from their bottom-right handle, and clamped only to the live viewport edges; only the initial spawn position should reserve extra top clearance in `em` units below the routed top bar
- resizing should allow the window to grow to the full available viewport area instead of stopping at a smaller fixed max width or height
- the bottom-right resize handle should stay visibly above iframe content and scrollbars as modal-owned chrome, using a small cut-corner treatment that matches the window shell while keeping the drag target larger than the visible backing
- minimizing a window should collapse it to a fixed `12em` width while anchoring the right edge in place so the minimize or restore button stays under the cursor
- window state must stay browser-local only; do not create backend state for browser windows, but do persist each open window's URL, position, size, minimized state, internal id, and stacking data in origin-local browser storage so reload or packaged-app relaunch can restore the same window set
- browser-window restore must reuse the same viewport-fit pass that runs on live viewport resize, so restored windows are clamped back onto the current screen and any adjusted geometry is immediately written back to browser-local storage
- focusing a browser window through any of its owned surfaces, including direct clicks into iframe or desktop webview page content, must raise that window to the top by assigning it the current highest browser-window z-index plus one
- new windows should spawn near the left edge of the centered router stage column, should default to approximately that column width instead of spanning the full wide-screen viewport, and should default to roughly `80vh` height capped at `100em`
- each window body must render a real `<x-browser controls="true" data-browser-id="browser-N">`; browser sessions render an iframe fallback inside that element, packaged desktop runs render a DOM-backed `<webview>` inside it, and the legacy native desktop path may map the same DOM element to a native browser view
- `<x-browser>` is placement-generic: any screen DOM, including a space widget renderer, may create `<x-browser src="google.com"></x-browser>` and the store must allocate a unique internal `browser-N` id, normalize the `src` like address-bar input, register it in the same browser-surface list as popup windows, and expose it through `space.browser` plus browser transient context
- `<x-browser controls="true">` must render the shared address bar with Back, Forward, Reload, and address controls; missing `controls` and explicit false values such as `controls="false"`, `controls="0"`, `controls="no"`, or `controls="off"` must render only the browser content
- the `<x-browser>` shell must consume the full available height of its placement, must keep the active inner browser surface filling both axes instead of inheriting the embedder element's default intrinsic height, and must keep the frame shell plus immediate rendered child hit-transparent where needed so iframe, `<webview>`, or native content receives direct page input
- packaged desktop runs must prefer a DOM-backed `<webview>` inside `<x-browser>` so clipping, rounded corners, resize chrome, inter-window stacking, and widget placement stay in the same compositor layer as the owning renderer DOM
- each iframe fallback must load `/mod/_core/web_browsing/browser-frame.html`, must carry `data-space-inject="/mod/_core/web_browsing/browser-frame-inject.js"` on the iframe element itself, and must use the same unique `browser-N` id and `name` pair as the owning browser surface
- iframe fallback startup must resolve that placeholder against the current app origin and must not let the iframe's transient pre-load `about:blank` state overwrite the intended local placeholder URL
- the title bar must expose a quick-add button on the left that opens another browser window and must use a non-plus restore icon on the minimize or restore control
- each non-minimized popup window must opt into the generic `<x-browser controls="true">` toolbar; in browser-mode sessions the placeholder page still loads first and the address or reload controls drive the iframe fallback even though injected navigation features remain inactive there
- toolbar pointer interaction must raise the owning browser modal without immediately refocusing the native browser surface, so the address field and toolbar buttons keep normal DOM focus inside packaged desktop runs
- this module must expose a narrow top-level `space.browser` runtime namespace for console and cross-feature access; its public agent-facing ids should be numeric ordinals such as `1` even though the internal DOM or store id remains `browser-1`, and that public namespace should mirror the current browser-surface list through `ids()`, `list()`, `count()`, `has(id)`, and `state(id)`, expose top-level `open(...)` and `create(...)` for new browser modals plus `close(id)` and `closeAll()` for any registered browser surface, expose `send(browserId, ...)`, a direct `evaluate(id, scriptOrPayload)` escape hatch for in-page JavaScript execution, convenience inspection wrappers `dom(id, ...)`, `content(id, ...)`, and `detail(id, ref, ...)`, direct navigation helpers keyed by browser id, simplified reference-targeted helpers such as `click(id, ref)`, `type(id, ref, value)`, `submit(id, ref)`, `typeSubmit(id, ref, value)`, and `scroll(id, ref)`, and the browser-operator-only `setLogLevel(level)` helper for adjusting app-side browser diagnostics; do not expose per-window handles, `current()`, `get(...)`, or a public `sync(...)` step in the agent-facing contract, because open or state or navigation or inspection helpers must settle internally and return fresh browser state snapshots while ref-targeted interaction helpers should return `{ action, state }` so callers can inspect action-level status flags such as `reacted`, `noObservedEffect`, `validationTextAdded`, `nearbyTextChanged`, or `domChanged`; navigation-capable helpers in particular must wait for an observed browser-side navigation or loading transition before they accept a settled snapshot, and while that handoff is still unobserved the store must not trust the old guest bridge or let it overwrite the optimistic destination with stale page state; address-entry helpers such as `open(...)`, `create({ url })`, inline `<x-browser src>`, and typed navigation should treat bare hosts like `novinky.cz` or `localhost:3000` as browser-address input rather than app-relative paths, promoting normal domains to `https://` and localhost or IP-style targets to `http://`; `open(...)`, `create(...)`, `state(...)`, inspection helpers, navigation helpers, interaction helpers, direct browser-surface focus, and `evaluate(...)` should all mark that browser as the current prompt-time browser-content source, and `typeSubmit(...)` should mean type into the field and then press Enter in that same field instead of forcing an immediate form submit path; `dom(...)` and `content(...)` should accept either `payload.selector` or `payload.selectors` for scoped reads, and the public API may still accept legacy `"browser-N"` strings as input, but snapshots should report numeric ids
- the browser store must keep each registered browser surface's latest `currentUrl` and `title` in browser-local state so owner-module prompt hooks can summarize open browser status without poking live iframe, webview, or native-view DOM directly
- the browser store must also remember the last exact browser instance targeted by agent-driven open or create or navigation or inspection or interaction helpers plus explicit `state(...)` checks or direct surface focus, but it must not fetch or cache page content at that interaction time; owner-module prompt hooks should fetch fresh `content` only while prompt transient sections are being built, and only if that remembered browser instance still exists instead of a later reused `browser-N` id
- all agent-facing `space.browser` functions must start through the shared `agent-function-availability.js` guard rather than open-coding runtime checks; that guard is intentionally generic, currently enforces the `native_app_only` requirement from the framework runtime context, must treat the packaged desktop `spaceDesktop.browser.available` bridge as authoritative even if a stale DOM runtime tag still says `browser`, and in standalone browser sessions must return a structured warning object whose message says `Browser functionality is currently only implemented in native apps.` while also logging that same plain-text warning to the console so blocked calls still surface a readable hint in execution output
- the outside helper at `/mod/_core/web_browsing/browser-frame-bridge.js` must expose a console-friendly `send(browserIdOrElement, type, payload)` request helper that resolves either a browser id or a DOM reference to an `<x-browser>`, inner iframe, or inner `<webview>` across the iframe fallback, desktop `<webview>`, or desktop-native bridge and returns the response payload
- every bridge envelope must be prefixed by the module-local `space.web_browsing.browser_frame` channel and must include `type` plus JSON-safe `payload`; request and response envelopes must also carry `requestId`, and responses should reuse the originating request `type`
- packaged desktop runs now use three related browser-side injection paths: the packaged frame preload still owns iframe `data-space-inject` activation for browser-mode iframes; the preferred desktop path renders a DOM `<webview>` inside `<x-browser>`, with Electron `will-attach-webview` forcing the owned preload and browser id metadata; and the legacy desktop-native `WebContentsView` path may still exist as a host fallback but must not be the primary browser-window surface while DOM clipping and stacking matter
- the packaged desktop browser preloads now form a small guest kernel: they must still run at document start, must force future `attachShadow(...)` calls open in the page's main world, must install the same cross-frame DOM helper in the top page and all subframes so shadow roots and iframe documents can be crawled later, and the desktop webview path must therefore keep `nodeIntegrationInSubFrames=true` for owned browser partitions only
- the desktop `<webview>` path must expose the same `__spaceBrowserEmbedTransport__` bridge from its preload, should publish explicit `sendEventName` plus `receiveEventName` DOM-event channels on that transport object so the guest page and the preload can exchange envelopes without relying on cross-world callbacks, may still keep callback helpers only as fallback compatibility, must inject the shared `/mod/_core/web_browsing/browser-frame-inject.js` bootstrap plus the phased guest runtime script lists returned by `browser-guest-runtime.js` into each guest page after `dom-ready`, must run any `preBootstrapScriptPaths` before the bootstrap source and any `postBootstrapScriptPaths` after it so handler-registration scripts see a live runtime registry, must treat that injection as document-scoped and reinstall it after every full guest-page navigation instead of only on the first placeholder page, must wait for guest handler-ready events before it starts retrying bridge-driven navigation-state sync, must also defer bridge-driven state reads during an unobserved navigation handoff so a still-alive old guest document cannot overwrite the returned state snapshot, and must keep toolbar navigation state fresh from both immediate embedder reads and bridge events
- desktop `<webview>` helper reads and fallback toolbar actions must tolerate pre-`dom-ready` guests without throwing; until Electron enables navigation methods, host-side state should fall back to the embedder `src` and retry after the normal `dom-ready` plus bridge sync path
- desktop `<webview>` markup must not bind guest `src` reactively from Alpine state; the store should set the initial `src` imperatively during registration and use bridge or fallback navigation methods for later changes so toolbar sync does not restart guest loads
- desktop `<webview>` registration must also patch only the active host shadow-root frame chain so Electron's internal guest iframe fills the full shell height, passive shadow-root siblings do not intercept pointer input, and direct pointer interaction with that browser surface should focus the webview before the guest handles input
- transparent or partially transparent remote pages should reveal a browser-default white backing inside the content shell rather than the modal's dark chrome
- desktop `_blank` and `window.open(...)` behavior must stay in-app: the injected runtime should emit `open_window` bridge events so the renderer opens a fresh `browser-N` modal instead of allowing a separate OS window
- the injected-side runtime bootstrap must expose a runtime registry with `registerMessageHandler(...)`, a shared `handleMessage(...)` dispatcher, event-send helpers, and the reusable DOM or content or action or navigation utilities needed by later guest runtime scripts; module-owned guest runtime scripts should register request handlers through that registry instead of hardcoding those registrations into the bootstrap
- the injected-side bridge bootstrap should log one positive guest-console line when the bridge runtime itself is ready, should emit one lightweight `__bridge_ready__` event back to the host immediately after that bootstrap succeeds so renderer-side bridge logs can confirm guest-to-host transport before any request runs, and the current core guest runtime script must register `ping`, `dom`, `content`, `detail`, `evaluate`, `click`, `type`, `submit`, `type_submit`, `scroll`, `navigation_state_get`, `location_navigate`, `history_back`, `history_forward`, and `location_reload`, must install navigation-state emission plus `_blank` or `window.open(...)` interception there, should log a second positive guest-console line when those core handlers are installed so devtools can confirm the page is listening, and later modules should extend that runtime by contributing more injected guest scripts through the guest-runtime builder seam rather than editing the bridge bootstrap directly
- the injected-side `dom` capture must prefer the document-start DOM helper when present so it can serialize open shadow roots and nested iframe documents from the top page down instead of stopping at light DOM only
- the injected-side `content` and ref-targeted action flow must also prefer that document-start DOM helper when present: `content` should assign typed refs from the helper-produced recursive page tree, should request the helper's content-oriented snapshot mode so readable markdown does not leak synthetic helper shells such as `space-browser-frame-document` or `space-browser-shadow-root`, should drop obvious serialized-HTML text blobs, escaped browser-helper wrapper text, stray full-document shells such as `<html>` or `<!DOCTYPE ...>`, raw `<style>/<script>` challenge payloads, long escaped or minified widget-script text, and other obviously non-visible content such as nodes hidden through `hidden`, `aria-hidden`, `display:none`, `visibility:hidden|collapse`, `content-visibility:hidden`, or fully transparent `opacity:0` ancestors from the readable output while preserving real fenced code blocks, and should default to omitting link destinations, reference-label quotes, and list bullets while keeping list indentation unless the caller explicitly opts into `payload.includeLinkUrls === true`, `payload.includeLabelQuotes === true`, `payload.includeListMarkers === true`, or disables indentation with `payload.includeListIndentation === false`; readable refs should use the lean typed-box form like `[link 7] Story`, `[button 9] Confirm`, `[image 11] Hero image`, or `[input text 12] Search placeholder=Hledat`, may prepend compact state or semantic tags such as `disabled`, `checked`, `selected`, `invalid`, `muted`, `warning`, `error`, `success`, or `primary` unless `includeStateTags === false` or `includeSemanticTags === false` disables them, and links or images with no human label should fall back to truncated URL text instead of a blank marker; if a page's Trusted Types policy blocks the helper-backed HTML reparse used for readable rendering, `content` should fall back to live DOM capture instead of failing the whole request; `detail` should return the resolved target DOM plus descriptor or semantic tags and richer state metadata including value-like fields when available, `detail` or `click` or `type` or `submit` or `type_submit` or `scroll` should resolve those refs back to helper-owned `{ frameChain, nodeId }` routes when the referenced element came from a nested frame or helper-backed snapshot instead of a live top-document node, `detail` should remain the low-token path for inspecting one link destination, image source, or DOM target on demand, and helper-backed ref actions should return the same action-effect contract as the live DOM path so callers can detect visible no-ops instead of looping
- helper-backed `content` capture must stay best-effort across nested frames: a slow, cross-origin, or broken iframe may be omitted from readable content, but it must not abort the whole top-level page capture
- helper-backed readable capture must not collapse structural containers such as dialogs, regions, groups, forms, or navigation landmarks into one broad typed ref just because page event metadata made the helper aware of that container; those containers should render their readable descendants while their real links, buttons, images, inputs, and generic controls keep their own refs
- selector-scoped `content({ selector: "..." })` and `content({ selectors: [...] })` calls should stay scoped all the way down when the DOM helper is present: the helper should serialize only the requested live targets instead of first snapshotting the full recursive page tree, so large pages with many ads or nested frames remain usable under the standalone harness and other short-lived debugging surfaces
- browser guest runtime injection attempts should still surface readable diagnostics in desktop devtools, the desktop webview embed path should still mirror guest `[space-browser...]` or preload diagnostics back into the main renderer console through `console-message` events, and injection failures in the renderer plus request-handler failures inside the guest page should log explicit `console.error(...)` diagnostics so desktop devtools can show why a browser page or helper request failed; however, app-side browser logging must now flow through the shared browser log-level helper, default to `error`, and only emit lower-severity bridge or preload or handshake noise after an explicit `space.browser.setLogLevel("warn"|"info"|"debug")` change at runtime
- the injected-side runtime must stay transport-agnostic inside the page: it should use iframe `postMessage` when no desktop bridge exists and should switch to the desktop preload transport when `__spaceBrowserEmbedTransport__` is present, so later browser-extension injection can reuse the same page-side request handlers
- this module may also extend `_core/onscreen_agent/llm.js/buildOnscreenAgentTransientSections`; the current browser-owned `open-browsers` hook should inject nothing when no browser surfaces are open, and otherwise append one brief section headed `currently open web browsers` whose content is the compact pipe-delimited list `browser id|url|title` followed by one row per open browser using the public numeric browser id
- the current browser-owned `last-interacted-browser-content` hook should inject nothing when no remembered browser exists, when that remembered browser surface has already closed, or when fresh `content` cannot be loaded from it during prompt construction; otherwise it should append one section headed `last interacted web browser` containing the compact `browser id|url|title` row for that remembered browser plus a `page content↓` block populated from the live `content` bridge response fetched at prompt-build time, retrying a few short settle-and-read passes there so prompt-time content usually appears without teaching the agent an explicit `sync(...)` step
- normal browser sessions must still leave the file named by `data-space-inject` inactive until a non-desktop browser-side injector exists
- the placeholder frame should keep a dark background, centered red engineer full-body artwork from `/mod/_core/visual/res/engineer/astronaut_red_no_bg.png`, no drop-shadow treatment on that artwork, and the same slow floating motion language used by the first-party chat and launcher astronaut treatments
- that shared placeholder page must resolve its runtime from the nearest existing framework `<x-context data-runtime="...">` marker when it is embedded in the browser-mode iframe fallback, should mirror that decision into its own hidden local `<x-context>` marker for consistency, and should fall back to the packaged-app copy when it is running as a standalone top-level guest page in the desktop webview path
- in browser runtime it should show the exact text `Embedded browser only works in native desktop apps for now.` above a latest-release button that reuses the login screen's `Download native App` label plus platform logos and opens `https://github.com/agent0ai/space-agent/releases/latest` in a new window
- in packaged app runtime it should instead show the active-development browser notice with the exact text `Embedded browser is in active development and may still have some limitations.` plus the short secondary hint `Use the address bar above to open a page, or tell Space Agent what to do.` with an extra blank-line worth of separation above that hint

## Development Guidance

- keep browser-overlay behavior self-contained here unless a stable menu or router seam changes
- prefer extending this module's store and component pair over adding ad hoc globals or shell patches
- keep the browser surface generic enough that later browsing logic can replace only the inner browsing engine or injection transport without rewriting the floating-window chrome
- if the overlay seam, menu-item order, browser id scheme, `<x-browser>` contract, toolbar contract, injected-runtime transport, `data-space-inject` contract, packaged-desktop browser bridge rules, or browser-frame bridge envelope or message types change, update this file, `/app/AGENTS.md`, `/packaging/AGENTS.md`, and the matching docs under `_core/documentation/docs/`
