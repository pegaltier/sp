# AGENTS

## Purpose

`packaging/` contains native app hosts and the packaging surface around them.

Keep this file scoped to native hosting and packaging behavior. Repo-wide packaging surface and install commands still belong in `/AGENTS.md`.

This is one of the five core docs. It owns the packaging subtree contract. If platform-specific packaging areas later grow their own `AGENTS.md` files, those local docs should own the detailed implementation guidance while this file stays focused on packaging-wide structure and principles.

Documentation is top priority for this area. After any change under `packaging/` or any packaging contract change owned here, update this file and the matching supplemental docs under `app/L0/_all/mod/_core/documentation/docs/` in the same session before finishing.

## Documentation Hierarchy

`/packaging/AGENTS.md` stays the packaging-wide doc until a host surface, platform subtree, or packaging helper area grows its own `AGENTS.md`.

If that happens:

- this file should keep packaging-wide structure, shared principles, and cross-platform rules
- the child doc should own the exact host, platform, bridge, or build-helper contract
- update both docs when a packaging-wide rule and a local implementation contract change together

## How To Document Packaging Child Docs

Future packaging docs should keep the same section spine:

- `Purpose`
- `Ownership`
- `Host Or Build Contract`
- `Platform Or Bridge Contract` when that distinction exists
- `Development Guidance`

Required coverage:

- the startup lifecycle between the native host and the local server runtime
- any preload or native bridge APIs exposed to browser code and who may call them
- packaging scripts, assets, metadata, and platform-specific files owned by that subtree
- what must stay thin host glue versus what must remain in `app/` or `server/`
- platform-specific divergence only where it is real; shared behavior should stay documented here

## Current State

`packaging/desktop/` holds the current Electron desktop host.

`packaging/desktop/preload.js` exposes the launcher-shell desktop bridge as `space` on the recovery-safe `/enter` and `/login` pages only, including packaged-runtime info, background update-check hooks, explicit download or install actions, and the packaged debug-only `debugReinstall(version?)` hook. The main app runtime keeps owning `window.space` everywhere else, so the preload must not publish a competing bridge on ordinary app routes; instead it now exposes the ordinary-app `spaceDesktop.browser` bridge for desktop browser embedding metadata, host events, and any legacy native-browser hooks. `packaging/desktop/frame-preload.js` is the packaged-app-only iframe preload split in two parts: in the top-level app frame it watches iframe `data-space-inject` markers, ensures each opt-in iframe is registered under a unique frame `name` plus host-visible `iframeId`, and registers those entries with the main process so Electron can inject validated module scripts into the matching child frames after load; in subframes it runs a document-start main-world `attachShadow(...)` override that forces future shadow roots open inside iframe documents. `packaging/desktop/browser-dom-helper.js` is the packaged desktop browser guest kernel installed into the page's main world at document start; it maintains the frame-local DOM helper state, recursively snapshots shadow roots plus nested iframe documents, assigns helper-owned node ids inside each frame, routes later detail or action requests back through the correct `frameChain`, and must log request failures to the guest-page console so desktop devtools can show frame-helper problems directly. Those helper-owned node ids are part of the readable-content contract too: any element that `content(...)` exposes as a typed ref, including plain `IMG` nodes and generic event-bound controls such as framework-owned clickable `div`s, must receive a helper node id so later `detail(...)` lookups succeed in packaged desktop mode. The helper must also serialize the compact state or semantic tags and live value text needed for lean descriptors such as `[disabled muted button N] Continue`, `[checked checkbox N] Email updates`, or `[input text N] Search value=Ethereum`. That helper now has two distinct snapshot modes: raw `dom` snapshots may still emit synthetic wrapper elements such as `space-browser-frame-document` and `space-browser-shadow-root` for debugging, while content-oriented snapshots must strip those helper shells, unwrap any stray helper-wrapper or whole-document shells that survive nested frame capture, inline only the useful nested body or shadow content so downstream readable markdown stays human- and agent-focused, and omit obviously non-visible nodes such as `hidden`, `aria-hidden`, `display:none`, `visibility:hidden|collapse`, `content-visibility:hidden`, or fully transparent `opacity:0` subtrees from the readable snapshot. When the caller supplies selector-scoped capture for either raw `dom` or content-oriented reads, through `selector` or `selectors`, that helper should serialize only the requested live targets instead of snapshotting the entire recursive page first, so short-timeout debugging surfaces stay responsive on ad-heavy pages. Content-oriented frame capture must also stay best-effort: if one nested iframe cannot be captured quickly or at all, the helper should omit that iframe's readable content instead of aborting the whole top-level capture. Helper-backed actions should return the same action-effect payloads as the live DOM path too, including visible-reaction flags such as `reacted`, `noObservedEffect`, `validationTextAdded`, and semantic hints for newly revealed text. `packaging/desktop/browser-webview-preload.js` is the preferred desktop browsing preload: it opens future shadow roots inside packaged `<webview>` guest pages, installs that same guest kernel into the top page and all subframes, and exposes the view-local `__spaceBrowserEmbedTransport__` bridge so the injected runtime can use the same request-response envelope contract there that the iframe path already uses for `ping`, `dom`, `content`, `detail`, `evaluate`, ref-targeted page actions, navigation requests, and host events. That transport should now publish explicit `sendEventName` and `receiveEventName` DOM-event channels for guest-page communication instead of relying on cross-world callback invocation, may keep callback helpers only as fallback compatibility, should emit lightweight `__preload_ready__` and `__preload_received__` debug events back to the host so the embedder can prove the preload is alive during transport debugging, and the guest bootstrap should emit one lightweight bridge-ready event back to the host as soon as the transport comes up. The renderer-side store should wait for those guest readiness events before it starts retrying navigation-state bridge calls, and it should mirror guest browser-debug console lines back through the webview `console-message` surface so the main renderer devtools show guest bootstrap state without opening a separate inspector. The desktop browser path should produce readable devtools diagnostics at every stage, but the app-side browser logger now gates those mirrored diagnostics through `_core/web_browsing`'s shared browser log level: packaged app runs default to `error` so routine preload or bridge chatter does not flood the renderer or the agent-facing framework, and operators can raise that threshold at runtime with `space.browser.setLogLevel("warn"|"info"|"debug")` when they need deeper transport tracing. `packaging/desktop/browser-preload.js` remains the legacy native-browser-view preload for the older `WebContentsView` path.

`browser-dom-helper.js` must not assign helper-owned action ids to structural containers such as dialogs, regions, groups, forms, or navigation landmarks merely because those containers carry focus or event metadata; the readable layer must see those as containers so consent dialogs and similar overlays keep their body text while real controls inside them keep refs.

`packaging/desktop/server_storage_paths.js` is the canonical packaged-desktop helper for choosing the stable packaged `userData` root across desktop rebrands and for locating writable server temp and auth fallback storage outside the installed bundle.

Standalone browser-bridge debugging belongs under `tests/browser_component_harness/`, not inside the real desktop host. `packaging/desktop/main.js` and `packaging/desktop/preload.js` must stay focused on production desktop behavior and should not grow test-driver IPC or scenario hooks for browser harness work.

`packaging/desktop/updater_artifacts.js` is the canonical packaged-desktop helper for updater cache ownership: it writes a marker before explicit restart-to-install handoff, then clears stale `pending/` payloads from the current and legacy updater cache roots on the next packaged launch while leaving reusable metadata such as blockmaps intact.

`packaging/desktop/updater_install_options.js` is the canonical packaged-desktop helper for updater log-path and stock NSIS-argument resolution. It owns the stable packaged updater log path `<userData>/logs/desktop-updater.log`, and the desktop main process appends best-effort diagnostics there before explicit restart-to-install.

`packaging/desktop/updater_debug_release.js` is the canonical packaged-desktop helper for packaged updater release-asset naming. It resolves release tags, platform metadata filenames, and canonical GitHub Release asset URLs, stages same-version reinstalls or specific-version downgrades for the debug bridge, and also owns the Windows arch-mismatch detection that lets the packaged host recover when a published Windows metadata file omits the current build's installer.

`packaging/package.json` holds packaging-only dependencies so the root install can stay lean, and packaging scripts must treat that file as the authoritative Electron version for native builds. The desktop builder may reuse the locally installed Electron distribution only when packaging for the current host platform; when cross-packaging from another host it must leave `electron-builder` free to resolve or download the correct target runtime instead of forcing the local host Electron bundle into a foreign platform build.

Runtime dependencies required by `packaging/desktop/main.js`, such as the Electron updater client, must live in the root `package.json` `dependencies` block so `electron-builder` copies them into the packaged app under `Contents/Resources/app/node_modules`.

The root `package.json` `build` block owns the Electron app entry, copied files, and `directories.app` selection. That config must keep the app directory pinned to the repo root because `app/package.json` exists only to mark the browser tree as an ES module package boundary, not to redefine the desktop host's app root. Because packaged desktop apps always relocate writable customware into `<userData>/customware`, that same build file set must exclude repo-local `app/L1/` and `app/L2/` content from the bundled app. The desktop host must also keep that packaged `userData` root stable across app-name rebrands by preferring any legacy runtime-owned directory that already contains `customware/` or `server/data/` before it starts a fresh current-name root. The Windows NSIS config in that same build block must pin an explicit stable `nsis.guid` once the app is released, keep any one-off legacy uninstall-key migration include under `packaging/platforms/windows/`, and recreate desktop shortcuts on reinstall so updater-driven repairs do not leave stale links behind.

Linux packaging metadata that is target-specific should live in the root `package.json` `build.linux` block, such as the explicit maintainer string used by Linux package formats, rather than forcing unrelated global `author` changes. `packaging/platforms/windows/installer.nsh` now owns the Windows NSIS migration hook that teaches installers built with the canonical GUID to recognize the accidental rebrand-era uninstall key during upgrades, plus the installer-side running-app wait loop and installer-progress logging that harden update-time reinstalls after `electron-updater` launches the stock NSIS installer.

That same root build config disables `npmRebuild` for desktop packaging. Packaged desktop builds rely on the native-or-isomorphic Git backend abstraction instead of Electron-time rebuilds of native Git addon dependencies.

`packaging/scripts/` holds packaging entrypoints and shared build helpers. Multiword operation entrypoints use object-first hyphen naming such as `host-package.js`, `macos-dev-build.js`, and `desktop-dev-run.js`. `desktop-builder.js` is the canonical wrapper around `electron-builder`, `release-version.js` is the canonical release-version helper for local and CI desktop builds, `release-metadata.js` owns shared updater-metadata parsing and serialization, `release-metadata-merge.js` owns the multi-arch updater-metadata merge step, and `release-assets-stage.js` stages the final GitHub Release upload set. `macos-dev-build.js` is the canonical zero-required-args wrapper for local macOS development packaging: it defaults to an unpacked `--dir` build, sets `SKIP_SIGNING=1` when that env var is otherwise unset, forwards optional arch or version overrides to the shared macOS packaging path, and is the intended entrypoint behind `npm run package:desktop:macos:dev`. Desktop version resolution must prefer explicit script input, release environment variables, an exact checked-out Git tag, and only then the root package version, so OS app metadata matches the release being bundled. That helper also derives the two-segment release version used for published GitHub asset names by stripping a redundant trailing `.0` patch from semver-like tags such as `v0.40.0`. Keep the root `package.json` version aligned with the current Git-derived release version because npm script banners and package-version fallback paths read that metadata directly.

`packaging/release-asset-filters.yaml` owns the editable public-installer upload allowlist for the desktop workflow. During publish, `release-assets-stage.js` stages only the kept public installers under the uniform release asset name `Space-Agent-<release version>-<platform>-<arch>.<extension>`, rewrites the updater metadata files `metadata-latest-windows.yml`, `metadata-latest-mac.yml`, `metadata-latest-linux.yml`, and `metadata-latest-linux-arm64.yml` so they point at those same canonical names plus the canonical macOS updater zip names `Space-Agent-<release version>-macos-<arch>-update.zip`, and removes duplicate builder-default asset names and blockmaps from the GitHub Release. Published public asset names use the two-segment release version when the semver patch is `0`, so tags such as `v0.40.0` upload as `Space-Agent-0.40-<platform>-<arch>.<extension>` while packaged app metadata keeps the full semver build version required by the desktop toolchain and user-facing updater labels collapse that same redundant trailing `.0` patch to the two-segment display version.

`packaging/resources/` holds shared packaging resources, including the canonical source artwork that desktop packaging can derive platform icons from.

`packaging/platforms/` holds OS-specific packaging assets and metadata, including derived app icons such as the Windows `.ico` file and the Linux icon-size set.

## Guidance

- avoid moving app logic into native hosts
- keep packaging automation in `packaging/scripts/`
- keep multiword packaging script filenames object-first so related entrypoints sort together
- keep platform-specific packaging details in `packaging/platforms/`
- keep release-staging verification data out of hidden scratch directories under the repo; do not create or commit paths such as `.tmp/` for packaging outputs, manifests, or generated binaries
- if packaging needs a checked-in fixture at all, keep it small, synthetic, and under an owned non-hidden test or documentation path instead of committing staged release artifacts or installer binaries
- keep local packaging workable without Apple credentials; the desktop builder consumes `SKIP_SIGNING=1` for macOS packaging runs and disables signing and notarization in that mode, and `npm run package:desktop:macos:dev` is the preferred zero-config local wrapper because it defaults that env var plus `--dir`; for signed local runs the shared builder also accepts the launcher-style `APPLE_PASSWORD` env var as an alias for `APPLE_APP_SPECIFIC_PASSWORD`
- add future mobile-specific hosts alongside `packaging/desktop/`
- when native host behavior, preload bridges, packaging assets, or packaging entrypoints change, update this file in the same session
